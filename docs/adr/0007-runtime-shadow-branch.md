# ADR-007: 运行时阴影分支代替编译期宏

## 状态
已采纳

## 背景
阴影采样逻辑的启用/禁用方式有两种选择：编译期通过 `#define USE_SHADOW` 条件编译，或运行时通过 `shadowIndex >= 0` 分支判断。若选择编译期宏，`ShaderProgramKey` 将从 5 维（32 种）扩展为 6 维（64 种），增加的维度没有其他用途。

## 决策
阴影采样逻辑始终编译在所有带光照（`useLights=true`）的着色器变体中。片元着色器通过运行时判断 `shadowIndex >= 0` 决定是否对当前光源执行阴影深度比对。

## 备选方案
- **编译期宏（`#define USE_SHADOW`）**：ShaderProgramKey 增加第 6 位 bool，变体扩展至 64 种。每个阴影宏变体都需重复编译完整的光照计算代码。
- **完全无阴影回退**：无阴影的光源在 C++ 侧跳过阴影贴图绑定，着色器不编译任何阴影代码。但用户无法在运行时动态开关阴影。

## 理由
1. **GPU 统一分支效率**：对于给定 Draw Call，所有片元的 `shadowIndex` 一致（场景中光源固定），GPU 分支预测器可完美处理——几乎零线程发散代价。
2. **变体数量减半**：从 64 种降为 32 种，减少了 Metal 着色器编译时间（尤其是首次启动时的 PSO 编译）。
3. **运行时灵活性**：用户可在不重新编译着色器的情况下，通过 `light->castShadow = false` 动态关闭某光源的阴影。
4. **简化 MSL 模板生成**：不需要在运行时拼接 `#define USE_SHADOW` 字符串，ShaderProgramKey → MSL 前缀的翻译逻辑保持简单。

## 影响
- 正：32 种变体上限，Metal 着色器函数缓存命中率高。
- 正：动态阴影开关无需重新编译 PSO。
- 负：片元着色器中始终包含阴影采样代码（即使场景无阴影），但分支预测保证无效路径的代价极低（ALU 指令缓存占用略有增加）。
- 负：无法通过预处理完全剥离阴影代码——对于极端受限场景（如数千个光源、无阴影需求）略有浪费，但此类场景不在 P2 考虑范围内。
